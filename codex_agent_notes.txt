================================================================================
  WHAT THIS PROJECT IS
================================================================================

SciLibrarian is an AI-powered knowledge management platform for research teams.
Its core feature is an AI librarian agent named Alexandria who helps researchers
ingest, organise, search, and stay current with academic literature.

Primary use case: the Australian AI Safety Institute managing a growing
library of AI safety papers, policy documents, model cards, and evaluations.

Key user workflows:
  1. Upload PDFs or paste URLs → Alexandria extracts text, generates summary + tags
  2. Browse the library by collection (hierarchical folders) or search full text
  3. Set up monitors (automated searches) → results go to Review Queue → approve
  4. Ask Alexandria questions in the chat panel → she searches the library + web
  5. Generate monthly digests → state-of-the-art synthesis of the library
  6. Watch Requests → plain-language descriptions of what you're looking for


================================================================================
  TECH STACK
================================================================================

Backend:
  - Python 3.12, FastAPI 0.115, SQLAlchemy 2.0 (async), asyncpg
  - PostgreSQL 16 (database, no vector store yet — pgvector column exists but
    embedding generation not yet wired end-to-end)
  - LiteLLM 1.56 (unified AI API — Claude, GPT-4o, Gemini, Ollama, vLLM)
  - For Ollama specifically: bypasses LiteLLM and calls /api/chat directly
    (reason: LiteLLM's ollama provider can't forward think=false, breaking
    qwen3/deepseek-r1 streaming — see docs/ollama-integration-investigation.md)
  - PyMuPDF (PDF text extraction), httpx, BeautifulSoup4, pandas, openpyxl,
    python-docx (multi-format ingestion)
  - APScheduler (in-process scheduler: monitors every 6h, digest monthly)
  - duckduckgo-search, imap-tools, aiosmtplib

Frontend:
  - React 18, Vite 6, Tailwind CSS 3
  - TanStack Query v5 (server state), React Router v6, Axios
  - Lucide icons, react-hot-toast, react-markdown

Infrastructure:
  - Docker Compose (db, backend, frontend, optional ollama via --profile ollama)
  - Ollama service mounts host model directory /usr/share/ollama/.ollama
    (avoids re-downloading models; 13 models including gemma4, qwen3.5, medgemma)
  - GPU passthrough: NVIDIA Container Toolkit, deploy.resources in compose
  - Default model: ollama/gemma4:latest (RTX 5090 available on dev machine)

================================================================================
  DATABASE SCHEMA (key points)
================================================================================

Tables: users, projects, collections, references, reference_tags,
        review_queue, search_monitors, digests, watch_requests

Schema management: NO Alembic. Uses SQLAlchemy create_all() + a _MIGRATIONS
list in database.py that runs ALTER TABLE IF NOT EXISTS on startup.
This is intentional (simple, self-healing) but means no migration history.

Key design choices:
  - 'references' is a reserved SQL word — must be double-quoted everywhere
  - projects → collections → references (hierarchical, all project-scoped)
  - All collections require project_id (added via migration, nullable)
  - CollectionOut MUST be built with _col_to_out() — NOT model_validate()
    because model_validate tries to access lazy SQLAlchemy relationships
    and crashes with MissingGreenlet error in async context
  - References have: is_starred (bool), read_status (unread/reading/read),
    notes (text), full_text (text), file_path (local storage only)

Auth: JWT, username+password only (no email required by design).
      Tokens stored in localStorage (frontend).


================================================================================
  KNOWN ISSUES AND ROUGH EDGES
================================================================================

1. NO ALEMBIC — schema changes are manual ALTER TABLE statements in database.py
   _MIGRATIONS list. This is fragile and doesn't track history. A proper
   Alembic setup is a recommended improvement.

2. EMBEDDINGS NOT WIRED — services/embeddings.py exists and pgvector dependency
   is installed, but the Reference model has no embedding column yet and
   get_embedding()/similarity_search() are never called. The infrastructure
   is there; the migration and wiring are TODO.

3. COLLECTION SCOPING INCOMPLETE — references have project_id but the list/search
   endpoints don't always filter by current project. A user can see references
   from other projects if they know the API directly.

4. NO RATE LIMITING — the API has no rate limiting middleware. Any authenticated
   user can hammer the LLM endpoints.

5. API KEYS STORED IN DB — per-project API key overrides (Anthropic/OpenAI/Gemini)
   are stored in plain text in project.settings JSON. Acceptable for internal
   tool, not for multi-tenant SaaS.

6. SCHEDULER RUNS IN-PROCESS — APScheduler runs inside the uvicorn process.
   With --workers 2 (production), each worker would run its own scheduler,
   causing duplicate monitor runs. Production should use a single worker or
   an external task queue (Celery/ARQ).

7. UPLOADS ON LOCAL FILESYSTEM — PDFs stored at /app/uploads (Docker volume).
   Production should move to S3/GCS. The file_path column stores absolute
   container paths which will break if the volume moves.

8. FULL TEXT TRUNCATED AT 50,000 CHARS — long PDFs are cut. The truncation
   happens in extract_pdf_text() and is intentional (LLM context limits) but
   means the search index and librarian don't see the full paper.

9. CORS IS WIDE OPEN IN DEVELOPMENT — allow_origins=["*"] in development mode.
   Tighten for production (update main.py CORS config).

10. FRONTEND USES LOCALSTORAGE FOR AUTH TOKEN — standard approach but vulnerable
    to XSS. HttpOnly cookies would be more secure.

11. useProject() HOOK RE-FETCHES PROJECTS INDEPENDENTLY — multiple components
    call useProject() which all issue a GET /projects query. TanStack Query
    caches these (same queryKey), so it's not N requests, but the hook is
    used in Sidebar AND individual pages, creating mild coupling.

12. CollectionOut HAS children FIELD — this field is only populated by the
    /tree endpoint. The flat /collections endpoint returns children=[] always.
    This is technically correct but confusing schema design.

13. REVIEW QUEUE FULL INGEST IS SLOW — each approved URL item runs the full
    ingestion pipeline (15-30s). Bulk approve with full ingest on 20 items
    will take 5-10 minutes and hold the HTTP connection. Should be a
    background job with progress streaming.


================================================================================
  AREAS TO FOCUS REVIEW ON
================================================================================

Priority 1 (correctness + reliability):
  - backend/app/routers/references.py: route ordering (/stats/summary must
    appear before /{ref_id} to prevent "stats" being parsed as int)
  - backend/app/services/llm.py: Ollama direct API path — error handling,
    timeout behaviour, what happens when Ollama is slow/unavailable
  - backend/app/services/ingestion.py: the new PDF-aware URL routing —
    edge cases (redirects, auth-gated PDFs, non-standard content-types)
  - backend/app/database.py: _MIGRATIONS running at every startup is safe
    (IF NOT EXISTS) but any error in a migration is silently swallowed

Priority 2 (security):
  - auth.py: JWT handling, token expiry, no refresh tokens
  - dependencies.py: current_user extraction
  - Any SQL injection vectors (SQLAlchemy ORM should prevent but verify)
  - File upload handling in references.py (path traversal, MIME spoofing)

Priority 3 (performance):
  - Collections tree endpoint: loads all children recursively with N+1 queries
    (one query per node). For deep/wide trees this degrades. A CTE-based
    recursive query would fix this.
  - proactive_search.py: Alexandria relevance filter adds one LLM call per
    monitor run. With slow Ollama models this could time out.
  - Library page: fetches up to 200 references client-side then filters.
    Should be server-side pagination + filtering.

Priority 4 (code quality):
  - Error handling consistency: some routers catch all exceptions silently,
    others propagate. A global exception handler would help.
  - Type hints: backend is mostly typed but some services have Optional without
    proper None checks
  - Frontend: no TypeScript (plain JSX). Some components are getting large
    (AddReferenceModal.jsx, ReferencePage.jsx) and could be split.


================================================================================
  HOW TO RUN (for testing during review)
================================================================================

Prerequisites: Docker, Docker Compose, NVIDIA Container Toolkit (for GPU)

Quick start:
  git clone git@github.com:mkuiper/SciLibrarian.git
  cd SciLibrarian
  cp .env.example .env
  # Edit .env: add ANTHROPIC_API_KEY or leave empty to use Ollama only
  docker-compose --profile ollama up --build
  # Frontend: http://localhost:5173
  # API docs: http://localhost:8000/docs

With Ollama (host models already downloaded at /usr/share/ollama/.ollama):
  OLLAMA_BASE_URL=http://ollama:11434 (default in .env)
  The ollama service mounts /usr/share/ollama/.ollama for instant model access

Without Ollama (cloud API only):
  docker-compose up --build
  Set ANTHROPIC_API_KEY or OPENAI_API_KEY in .env

Register at http://localhost:5173/register (no email needed, just username+password)
Create a project — Alexandria will design your collection structure using the
configured model.

Run tests manually:
  Backend API: http://localhost:8000/docs (Swagger UI)
  Health check: curl http://localhost:8000/health


================================================================================
  DESIGN PHILOSOPHY (why things are the way they are)
================================================================================

Full rationale in docs/design-decisions.md. Key points:

- Bypass LiteLLM for Ollama: LiteLLM's ollama provider can't pass think=false,
  causing streaming failures on qwen3/deepseek-r1 models. Direct /api/chat works.

- No Alembic: chosen for simplicity (ALTER TABLE IF NOT EXISTS is idempotent).
  Tech debt that should be addressed before any team collaboration on schema.

- No email required for auth: privacy decision. Username+password only.

- Human review queue: all monitor discoveries require approval before entering
  the library. Quality over volume — automated ingestion without review erodes
  trust in the library.

- Multi-provider via LiteLLM: allows the same codebase to work with Claude
  (cloud), GPT-4o (cloud), Gemini (cloud), or fully local Ollama models.
  Default is ollama/gemma4:latest since the dev machine has an RTX 5090.

- PDF-aware URL ingestion: arXiv URLs from monitors point to abstract pages or
  PDFs. The ingestion layer auto-detects and routes to the correct pipeline.
  History: this was a bug for several days — references from monitors had
  0 text until this was fixed.
