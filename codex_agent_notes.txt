================================================================================
  POTENTIAL ISSUES TO INVESTIGATE
================================================================================

1. FTS fallback on stopword-only queries
   plainto_tsquery('english', 'the a is') returns an empty tsquery which matches
   nothing. The current code returns an empty result set rather than falling back
   to a full list. Whether this is correct depends on UX intent.
   File: backend/app/services/search.py

2. _find_duplicate() empty-title edge case
   The function skips the title check when norm_title is empty, but the URL check
   still runs. A call with url=None and title="" returns None safely. Verify
   the from-url endpoint doesn't trigger a false 409 when a URL normalises to ""
   or when the URL dedup check fires before meta_title is known.
   File: backend/app/routers/references.py (lines around the two dedup calls)

3. test-key endpoint: key stored in request body, not header
   The API key is POSTed as a JSON body field. It will appear in backend access
   logs (request body is not typically logged by uvicorn, but is by some
   middleware). Verify no logging middleware captures it.
   File: backend/app/routers/config.py (KeyTestRequest)

4. Scheduler state detection uses magic integer constant
   scheduler_info["paused"] = sched.state == 2 relies on APScheduler's internal
   STATE_PAUSED = 2. This is not a public constant. Should use
   apscheduler.schedulers.base.STATE_PAUSED or check the import.
   File: backend/app/routers/config.py (~line 195)

5. Precision @property on Pydantic model
   SearchMonitorOut has a precision @property. Pydantic v2 does not serialise
   @property attributes — it will not appear in JSON responses unless explicitly
   included as a computed_field. The Monitors page calculates precision from
   approve_count and reject_count directly in JSX, so this is not currently
   broken, but the @property is dead code and may mislead future callers.
   File: backend/app/schemas/search_monitor.py

6. Monitor stat increment: no transaction isolation
   In review.py decide(), the monitor is fetched, incremented, and flushed in
   the same transaction as the queue item status update. If two concurrent
   approvals from the same monitor arrive simultaneously, both could read
   approve_count=5 and both write approve_count=6 (lost update). For low-
   concurrency internal use this is acceptable, but worth noting.
   File: backend/app/routers/review.py

7. Radar endpoint: inline imports inside function body
   project_radar() does `from app.models... import ...` and
   `from datetime import datetime, timezone` inside the function. These are
   deferred imports that work correctly but are inconsistent with the rest of
   the codebase (top-level imports). Minor style issue.
   File: backend/app/routers/projects.py

8. _dir_stats() blocks the event loop
   The os.walk call in config.py is synchronous and will block the async event
   loop for large upload directories. For small libraries this is fine; for
   large ones it could cause noticeable latency on the /config/system request.
   Fix: run in a thread pool via asyncio.to_thread().
   File: backend/app/routers/config.py (_dir_stats function)

9. Evidence trail parsing: model may omit or misformat Sources section
   The frontend parseSourcesFromResponse() uses lastIndexOf('\n### Sources\n').
   If the model writes "### Sources" without a blank line before it, or writes
   it mid-response rather than at the end, parsing silently returns no sources.
   This is cosmetic (no functionality broken) but could confuse users who see
   an answer with citations but no Sources panel.
   File: frontend/src/components/LibrarianPanel.jsx

10. Library.jsx: 'important' filter sends read_status=important but also has
    a separate filterUnread toggle sending read_status=unread. The two are
    mutually exclusive in the UI (filterImportant overrides filterUnread in
    the ternary), but the filterUnread state is not cleared when filterImportant
    is set. Cosmetically inconsistent.
    File: frontend/src/pages/Library.jsx


================================================================================
  KNOWN PERSISTENT ISSUES (carried from previous review)
================================================================================

A. Direct reference endpoint ownership still not enforced.
   GET/PATCH/DELETE /references/{id} fetch by id only; any authenticated user
   can read or modify any reference. The project_id and created_by fields exist
   but are not checked. This remains the highest-priority security gap.

B. list_projects returns all projects for all authenticated users.
   No team/membership model exists yet.

C. Bulk approve is synchronous HTTP. A 20-item queue with full ingest takes
   5-10 minutes. Should become a background job.

D. Scheduler runs in-process. With uvicorn --workers 2 (production docker-compose)
   both workers run a scheduler, causing duplicate monitor/digest runs.

E. API keys stored in plain text in project.settings JSON column.

================================================================================
  DESIGN DECISIONS (for context on choices made)
================================================================================

Full rationale in docs/design-decisions.md. Key decisions relevant to new code:

- FTS over ILIKE: plainto_tsquery chosen over to_tsquery for robustness (handles
  multi-word phrases, ignores stopwords). The GIN index expression must match
  the query expression exactly for the index to be used.

- Findings in extra_metadata.findings: not new columns. Allows schema evolution
  without migration cost. The fields (main_finding, method, limitations) may
  expand in future cycles.

- Evidence trails via model self-reporting (### Sources section): simpler than
  structured streaming metadata at the cost of reliability. Model may omit or
  misformat. Documented limitation.

- Dedup returns 409 with existing_id: allows frontend to redirect rather than
  silently swallow the error. Project-scoped so same paper can exist in two
  projects.
